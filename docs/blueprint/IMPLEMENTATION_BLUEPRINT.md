# IMPLEMENTATION BLUEPRINT

## 1. 项目定位
构建一个可持续演进的轻量化金融数据中心 MVP：
- 起点：本地 SQLite + 文件分层（raw/staged/mart）
- 目标：平滑演进到 PostgreSQL + FastAPI + 前台查询
- 首先聚焦组合层（Portfolio Layer）最小可用能力

## 2. 设计原则
1. **分阶段交付**：先打通最小闭环，再扩展复杂能力。
2. **边界清晰**：组合层、持仓层、交易层、行情层分层建设。
3. **可迁移性优先**：Schema 避免绑定 SQLite 专属能力，便于迁移 PostgreSQL。
4. **可审计可追溯**：引入 data_batch / data_issue_log 管理批次与数据质量。
5. **文档驱动施工**：HANDOFF 与 Blueprint 持续维护为 SSOT/方向锚点。

## 3. 阶段边界

### Phase 1A（当前）
- 范围：组合层基础主数据 + 日频净值 + 组合级指标 + 数据批次与质量日志
- 存储：SQLite
- 能力：建表、初始化、基础 smoke test
- 不包含：持仓、交易、行情接入、API、前端、复杂指标

### Phase 1B（下一阶段）
- 扩展：
  - 组合层数据装载流程（样例数据）
  - 数据校验与问题日志自动落盘
  - 基础查询服务接口（可先命令行/脚本）
- 仍不包含：真实生产联调与复杂分析引擎

### Phase 2（演进阶段）
- 迁移：SQLite + Parquet 原型升级为 PostgreSQL 主库
- 服务化：FastAPI 接口层
- 应用化：前台查询与管理页面
- 工程化：权限、监控、任务调度与 CI/CD

## 4. 数据分层策略
- `data/raw/`：原始落地文件（不改写）
- `data/staged/`：清洗后结构化中间层
- `data/mart/`：面向查询/消费的结果层
- `data/sample/`：演示和测试样例数据

数据库承载结构化事实与维表；文件分层承载数据交换与处理过程。

## 5. 技术路线
1. **当前**：Python + SQLAlchemy + SQLite
2. **过渡**：SQLite（开发）+ Parquet（批处理中间产物）
3. **目标**：PostgreSQL（核心库）+ FastAPI（服务层）

迁移原则：模型字段与约束设计尽量使用通用 SQL 概念，避免 SQLite 专有特性锁定。

## 6. 市场数据分层策略（规划，非当前实现）
- L0：外部行情源原始快照（文件/对象存储）
- L1：标准化行情事实表（日线/分钟线）
- L2：面向组合消费的聚合指标（收益、波动、回撤等）

> 当前 Phase 1A 不实现行情接入，仅预留架构位置。

## 7. 工程推进原则
1. 每轮施工必须有“可执行任务包 + 可验证命令 + 执行报告”。
2. 每轮小步提交，保持测试可运行。
3. 文档与模型同步更新，避免“代码真相”与“文档真相”分裂。
4. 先保障可维护性与可迁移性，再做性能优化。
